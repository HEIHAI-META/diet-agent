# 进食检测规则说明

基于 [eating-detector-demo.html](eating-detector-demo.html) 的当前实现。算法为**三因素加权占比**：每个因素直接贡献若干百分点，相加得到进食概率 P(Eating)，上限 100%。不再使用贝叶斯融合、log-odds、sigmoid，也不区分先验/后验。

---

## 1. 因素总览

只保留三类信号：**饭点时段**（时间上下文）、**符合历史习惯**（个人规律）、**听到「吃饭了」类确认语**（语义证据）。地点、停留、动作、心率等一律不计入。

| 因素 | 取值 | 贡献（百分点）|
|---|---|---|
| 饭点时段 | 非饭点 / 接近 / 饭点中 | 0 / 25 / 40 |
| 符合历史习惯 | 是 / 否 | 10 / 0 |
| 语音语义档位 | 未命中 / 即将吃 / 正在吃 | 0 / 50 / 75 |

**单个因素贡献排序**：正在吃语义 75 > 饭点中 40 > 即将吃语义 50 > 接近饭点 25 > 历史 10。

> **设计原则**：语音语义为**主导且可单独触发**——命中"正在吃"档（75）即使非饭点、不符合历史，P 仍达 75%，越过 70% 阈值；命中"即将吃"档（50）须配合饭点或历史才能越过阈值（如饭点中 40 + 即将吃 50 = 90%）。饭点与历史为上下文加分，单独或相加都不触发（饭点中+历史 上限 50%）。同一语音事件只取最高匹配档位，不叠加。多因素同时为真时直接相加，超过 100% 封顶为 100%。

---

## 2. 因素说明

### 饭点时段 mealTime
| 取值 | 贡献 | 含义 |
|---|---|---|
| 非饭点 none | 0 | 不加分 |
| 接近饭点 near | 25 | 饭点前后约 1 小时 |
| 饭点中 in | 40 | 7–9 / 11–13 / 17–19 点 |

真机模式按当前系统时间自动判定。

### 符合历史习惯 history
勾选才计 +10，未勾选 0。

### 语音语义档位 speechLevel
GPT5.5 对每句转写文字输出三档之一：`none / about_to_eat / eating`，取最高档贡献。

| 档位 | 贡献 | 判定条件 |
|---|---|---|
| `none` | 0 | 无进食相关语义 |
| `about_to_eat`（即将吃） | +50 | **空间转移指令 + 食物名词**同时出现：把孩子从当前活动召唤到餐桌，且对话中出现食物名词；例："关掉平板，洗手来吃饭！"、"快上桌，排骨熟了！"、"把玩具放下，过来坐好，今天有鱼。" |
| `eating`（正在吃） | +75 | **进食动作 + 食物实体 + 第二人称**三者同时出现：对话中明确有喂食/催食/纠正进食行为；例："宝宝把这块肉吃掉！"、"还剩两口，快吃！"、"别撒了，夹口菜。" |

同一语音事件只取最高命中档，不叠加。非饭点且无历史时，`about_to_eat` 档（50）不能单独越过阈值（需配合 饭点中 40 或 接近饭点 25 + 历史 10）；`eating` 档（75）单独即可越过阈值。

> 摄像头场景属 B 段，需用户同意后才采集，A 段不接入。
> 心率、动作、地点、停留等信号已移除，不计入 P(Eating)。

真机模式下，语音语义的派生路径：
- 浏览器 **Web Speech API** 持续转写麦克风语音（zh-CN，需 Chrome/Edge/Safari）。
- 每句话结束时，把转写文字 POST 到 `/eating-check`；服务端调 GPT5.5 判断该句属于 `none / about_to_eat / eating`（转写错别字按发音近似判断）。
- 命中 `about_to_eat` 或 `eating` 时更新 `speechLevel`，90 秒内无新命中则衰减至 `none`。

> 该网关（`/openai-compatible/v1/chat/completions`）不支持内联音频（`input_audio` 被拒，无 STT 端点），故走"浏览器转写 → 服务端文本判断"，GPT5.5 只收文本。

---

## 3. 计算方法

各因素贡献直接相加，超过 100% 封顶：

```
P(Eating) = min(100, 饭点 + 历史 + 语音确认)
```

1. 逐项累加各**为真**因素的贡献，得到合计 `raw`。
2. `raw` 可能超过 100（如饭点中 40 + 历史 10 + 语音 75 = 125），取 `min(100, raw)` 作为 P(Eating)。

> 与旧版贝叶斯的区别：不再有先验概率 P₀、似然合计、后验 log-odds 等中间量；因素之间是加法关系，不是 log-odds 空间的线性叠加。阈值比较直接用百分比。

---

## 4. 阈值与触发

- 阈值固定 **70%**，不可切换。
- 触发条件：`P ≥ 70%` 且 `armed` 且 当前无待回复弹窗。
- 触发后解除武装；`P < 67%`（70% − 3%）才重新武装（3% 滞回，避免抖动反复弹窗）。
- 触发后弹窗询问实际情况（正在吃饭 / 没在 / 刚吃完 / 即将吃）+ 备注，记录存 localStorage，可导出 JSON。

---

## 5. 算例

**`eating` 档单独触发（非饭点 + 不符合历史）**：

- 场景："宝宝把这块肉吃掉！"（非饭点）
- 合计：0 + 0 + 75 = 75
- **P = 75%** → 触发 ✓

**`about_to_eat` 档 + 饭点中**：

- 场景："关掉平板，洗手来吃饭，今天有排骨！"（饭点中）
- 合计：40 + 0 + 50 = 90
- **P = 90%** → 触发 ✓

**`about_to_eat` 档单独（非饭点 + 不符合历史）**：

- 场景：非饭点时大人说"去洗手，今天有鱼"
- 合计：0 + 0 + 50 = 50
- **P = 50%** → 不触发 ✓（需配合饭点才能越阈）

**饭点中 + 历史（无语音语义）**：

- 合计：40 + 10 + 0 = 50
- **P = 50%** → 低于 70%，不触发 ✓（纯上下文不过阈值）

**午餐场景末态**：饭点中 + 历史 + `eating` 档

- 合计：40 + 10 + 75 = 125
- **P = min(100, 125) = 100%** → 触发

说明：`eating` 档（75）是唯一能单独越过阈值的语义信号；`about_to_eat` 档（50）须配合饭点才能越阈，在非饭点噪声场景（看美食视频、过家家）下不误触发。饭点与历史仍作为上下文加分，单独不过阈值。

---

## 6. 提示频率

为避免一次进餐内反复提示，触发后进入**冷却期**：冷却期内即使 P 再次越过阈值也不会触发。

- 真机冷却 **30 分钟**；模拟模式 15 秒（便于测试）。
- 冷却到期后，须 P 再次从低位升到 ≥70% 才会触发下一次。
- 仍保留 3% 滞回（P<67% 才重新武装）和待回复阻塞（弹窗未关期间不触发）。
- 重置 / 切换模式 / 重播场景 会清空冷却。

效果：一次进餐大致只提示 1 次。若进餐时长超过冷却窗口（如超过 30 分钟），可能再提示 1 次；可调大 `COOLDOWN_MS` 延长冷却。

---

## 7. 触发后流程：提醒 → 确认 → 拍照

P ≥ 70% 触发后，按以下顺序执行：

1. **震动提醒**：设备震动 3 下（模式 180ms 振 − 120ms 停 − 180ms 振 − 120ms 停 − 180ms 振）。
   - 浏览器 `navigator.vibrate`；iOS 不支持该 API，静默跳过。
2. **语音播报**：合成语音朗读"是在吃东西吗？拍张照片记录一下吧"（zh-CN，语速 0.95）。
   - 浏览器 `SpeechSynthesis`；iOS 需用户手势触发，程序化触发可能不发声，Android 一般可用。
3. **弹窗确认**：保留原弹窗，四选一（正在吃饭 / 没在 / 刚吃完 / 即将吃）+ 备注。
4. **拍照**：仅当选"正在吃饭"时，调起系统相机（`<input type=file accept=image/* capture=environment>`，后置）拍一张。
   - 照片在本次会话内可预览（触发记录行显示缩略图，点击可看大图）；
   - 触发记录里持久化"已拍照"标记（`photoTaken`），**不存原图**，避免 localStorage 配额问题；
   - 调摄像头走浏览器权限弹窗，作为同意门。
5. **冷却**：触发即进入 30 分钟冷却（见第 6 节），与上述流程独立。

> 设备/原生端实现时：震动、TTS、相机走各平台原生 API；建议拍照前有显式同意门，照片按合规要求存储或上传。
