# 项目总结 Prompt · 饮食Agent MVP

> 本文档是项目的「上下文 prompt」——把一个新接手的 AI agent 当作刚走进房间、对项目一无所知的同事，喂给它读完即可上手。指令语气，第二人称。

---

## 你的角色

你是一个加入「饮食Agent」项目的工程协作者。这是一个面向 **4–10 岁儿童**的饮食追踪 demo：孩子佩戴无屏手环，系统在合适时机低打扰地确认孩子是否在吃，确认后引导拍照记录食物，生成餐盘结构与营养小科普；家长端同步日报/照片墙/周报，可对低置信度记录做确认修正。

你的任务是理解项目全貌、遵守既定规则与护栏、在用户给需求时正确地改代码。

## 两句话定位

- **A段** (`饮食A段/`)：进食检测的**单文件原型**（`eating-detector-demo.html` + `进食检测规则说明.md`），只做模块一的 P(Eating) 引擎，验证「饭点+历史+语音确认」加法模型。
- **B段** (`饮食B段/`)：**完整应用**。Vite + React 18 + TypeScript，设备端与家长端双视图，后端是跑在 dev server 内的 Vite 插件（`/api/*`），代理到 TAL GPT-5.5 网关（OpenAI 兼容）。B 段的模块一引擎移植自 A 段。

## 模块一规则（P(Eating) 引擎）

**加法模型**：`P(Eating) = min(100, 饭点 + GPS)`

| 信号 | 取值与贡献 |
|---|---|
| 饭点时段（触发依据） | 非饭点 0 / 接近饭点 40 / 饭点中 70 |
| GPS 就餐场景 | 非就餐场景 0 / 饭店・食堂等就餐场景 15 |

- **触发阈值 P ≥ 70%**；**重新武装阈值 P < 67%**（须先跌破 67 再升回 70 才允许下一轮，防抖动）。
- **触发前 3 条件全满足才提醒**：P≥70% ∧ 非冷却 ∧ armed。

**防打扰机制**：抬腕唤醒门（短震后等抬腕 8-10s，未抬腕 AI 不说话+记未唤醒+冷却）/ 30 分钟冷却（live 30min、sim 15s）/ going_to_eat 短等待 8-10min（sim 10s）后二次轻提醒 / 同一进食周期最多 2 次主动提醒 / 重新武装。
- **冷却在确认流程结束时设置**（`enterCooldown`），不是触发时——对齐 PRD「本轮流程结束→进/刷新冷却」。

**被动入口完整流程**：

```
触发条件全满足（P≥70% ∧ 非冷却 ∧ armed）
  ↓
手环短震
  ↓
等待抬腕唤醒（8–10 秒）
  ├─ 未抬腕 → AI 不说话；记「未唤醒」；计入本轮提醒次数；进 30min 冷却
  └─ 抬腕 → AI 第一问：「你现在是在吃东西吗？」
              ↓
       /api/interpret-reply 判 6 类意图
              │
              ├─ eating（在吃）
              │     → AI 第二问：「要不要拍一下帮你记录？」
              │       ├─ 同意 → 调起摄像头 → [进拍照执行链路] → 刷新 30min 冷却
              │       └─ 拒绝 → 记「在吃未拍照」
              │               → AI 追问：「你吃的什么呀？愿意和我说说吗？」
              │               ├─ 愿意口述 → 语音补录 → 记「语音补录，置信度低」→ 进 30min 冷却
              │               └─ 拒绝 / 无回复 → 进 30min 冷却
              │
              ├─ finished（吃过了 / 刚吃完）
              │     → AI 问「吃了啥」（一轮）→ 孩子口述 → 记「语音补录」
              │       → 待家长确认，不计趋势 → 进 30min 冷却
              │
              ├─ going_to_eat（正要吃 / 马上吃）
              │     → AI：「等你开始吃的时候可以告诉我，我可以帮你记录呦」
              │     → 进 8–10min 短等待（sim 10s）；之后 AI 重新判断语义：
              │       ├─ 有新进食语义 且 本轮提醒次数 < 2
              │       │     → 二次轻提醒：「你现在开始吃了吗？要拍一下食物吗？」
              │       │       孩子回复 → /api/interpret-reply 再判：
              │       │         ├─ eating   → 调起摄像头 → [进拍照执行链路] → 30min 冷却
              │       │         ├─ finished → 语音补录 → 30min 冷却
              │       │         └─ 否认 / ambiguous / no_response → 30min 冷却
              │       ├─ 无新进食语义 → 不提醒；进 30min 冷却
              │       └─ 孩子明确说没吃 → 不提醒；进 30min 冷却
              │
              ├─ not_eating（没在吃）→ 记「没在吃」→ 进 30min 冷却
              │
              └─ ambiguous / no_response → 进 30min 冷却
```

**主动入口（独立路径，优先级高于自动触发）**：

```
孩子主动说话（「帮我拍照」/「帮我记录一下」/「我吃了**」等）
  ↓
不卡 P(Eating) / armed / 冷却 / 提醒次数；不走震动→抬腕链
  ↓
默认走拍照路径，直接调起摄像头
设备：「好呀，摄像头已打开，请说「开拍」进行拍照」
  ↓
等待孩子回应：
  ├─ 说「开拍」→ 拍照 → [进拍照执行链路]
  │     AI 判质量：
  │       ├─ 高置信度
  │       │     → 设备：「好的，我记下来了」
  │       │     → 记「置信度高」→ 进食物分类（见下）→ 刷新 30min 冷却
  │       └─ 低置信度
  │             → 设备：「不太清晰，我们再拍一张，请说「开拍」」→ 补拍 1 次
  │               AI 再判：
  │                 ├─ 高置信度 → 设备：「好的，我记下来了」→ 记「置信度高」→ 进食物分类 → 刷新 30min 冷却
  │                 └─ 仍低置信度 → 设备：「还是不太清，不过我先记下了，回家让爸爸妈妈确认一下」
  │                               → 记「照片，置信度低」→ 刷新 30min 冷却（不走分类，流程结束）
  │
  └─ 说「不想拍照」/ 直接口述「我吃了**」
        → 不拍照，直接记录
        → 设备：「好的，我记下来了，回家让爸爸妈妈确认一下」
        → 记「语音补录，置信度低」→ 刷新 30min 冷却（不走分类，流程结束）

【食物分类判断】（仅置信度高时触发）
  ├─ 水果 + 正餐类 → 营养小科普
  └─ 零食饮料类 → 零食劝导
```

**拍照执行链路（三类入口统一，进此链路即走同一条路）**：

```
AI 语音引导摆盘（「从上面拍一下整个盘子，会看得更清楚哦」）
  ↓
屏幕倒数 3-2-1
  ↓
AI 自动拍摄（无孩子操作）
  ↓
LLM 判质量 → /api/analyze-intake
  ├─ 高置信度 → 食物识别 → 餐盘结构 → [营养小科普判断] → 同步家长端 → 流程结束
  └─ 低置信度 且 未补拍
        → AI 语音引导调整（「稍微拿远一点点，我想看清整个盘子」）
          ↓
        补拍 1 次（不另算打扰、不额外刷新冷却）
          ↓
        再判
          ├─ 高置信度 → 进识别下游（同上）→ 流程结束
          └─ 仍低置信度 → 降级「待确认记录」（照片保留给家长查看/修正）→ 流程结束
```

## 模块二规则（Recording）

**拍照执行链路（三类入口统一）**：被动 eating / 二次确认 / 主动拍照进**同一条**链路（引导摆盘→屏幕倒数 3-2-1→AI 自动拍，无孩子点按钮→LLM 判质量+置信度）：
- 高 → 进识别下游（餐盘结构 + 食物分类判断：水果+正餐类→营养小科普；零食饮料类→零食劝导）
- 低 且未补拍 → AI「不太清晰，我们再拍一张，请说「开拍」」 → 补拍 1 次 → 再判；仍低 → 降级「待确认」（照片保留给家长）；主动入口下仍低时设备说「还是不太清，不过我先记下了，回家让爸爸妈妈确认一下」
- 主动入口说「不想拍照」/直接口述 → 跳过拍照，记「语音补录，置信度低」，设备说「好的，我记下来了，回家让爸爸妈妈确认一下」，不走食物分类
- **仅置信度高时才走食物分类（营养小科普/零食劝导）；置信度低流程直接结束**
- **补拍不另算打扰、不额外刷新冷却**（冷却已在分支 finalize 时设）

**置信度**：高=俯拍全盘+清晰+主体完整+可识别 → 计趋势/生成结构/生成家长建议/触发科普；低（允许补拍 1 次）= 模糊/不完整/光线差/遮挡/主体不明/仅语音 → 待家长确认/不计趋势/不生成结构/不播科普。

**餐盘结构**：只评估 蔬菜≈1/2、主食≈1/4、肉蛋豆≈1/4，输出 偏少/适中/偏多（仅家长端可见）。水果/汤/零食独立记录，不参与比例。食物类别：蔬菜/主食/肉蛋豆/水果/汤/零食。

**营养小科普**：触发=正餐+有照片+高置信度+已识别具体食物+本餐未播+当天同食物未播+当天<3+非糖果/膨化/含糖饮料+LLM 安全。频次护栏=每正餐≤1、每天≤3、同一具体食物当天不重复（类别可重复）。LLM 规范：一句话、15-30 秒、小伙伴型；禁止评价本餐/克数卡路里/「吃了会长高变聪明」「不吃会生病」等。家长端只展示播放次数，不展示文案。

**零食记录与劝导**：零食与正餐**同走 `analyze-intake` 识别链路，按 `kind` 自然分流，不做独立入口**。零食独立记录、不归三餐、不参与餐盘结构、不生成科普。**劝导**仅糖果/膨化食品/含糖饮料，每天≤1（`/api/persuade-snack`），温和不批评/限制/恐吓。

## 家长端

- **日报**（默认 21:00 生成，可改）：日期概况 / 正餐详情（时间·图·食物名·置信度·结构标签·修改）/ 零食饮料 / 数据完整度 X/3 餐（标未记录/待确认/低置信度未计趋势）/ 今日营养评估（只定性不定量：星级 1-5 可半星失败默认 3 / 总评≤30 字 / 专业分析 2-3 句≤120 字 / 四维度 蔬菜·主食·肉蛋豆·添加糖·油脂 / 不良摄入清单 真实食物名+≤20 字原因 / 建议 2-3 条每条≤30 字）。
- **照片墙**：正餐/零食/饮料照片墙，新拍实时同步。
- **周报**：近 7 天有效记录；蔬菜不足 X/天（≥3 天高亮「建议密切关注」）+ 零食累计次数；异常日剔除。
- **配置**：营养小科普开关 / 晚间报告时间 / 异常日标注。
- **低置信度与 48h 归档**：低置信度按时间归入对应餐次带「待确认」badge；家长修改→置信度变「已确认」纳入趋势；超 48h 未处理（按模拟时钟判）→自动归档「未确认记录」不计趋势，仍可查看/修改。
- **双记录保留**：每条记录同时保留 AI 原始识别 + 家长确认值，家长修改不覆盖原始；日报/周报/趋势优先用家长确认记录。
- **趋势口径 = `confirmed`**：高置信度与家长确认的低置信度均计趋势；未确认/归档不计。

## 技术栈与架构

- **前端**：React 18 + TypeScript + Vite 5；状态 Zustand（persist 持久化测试历史到 localStorage）；图表 Recharts；语音 Web Speech API；HTTPS `@vitejs/plugin-basic-ssl`。
- **后端**：Vite 插件（`server/plugin.ts`）在 dev server 挂 `/api/*` 中间件，代理到 LLM 网关；API key 留后端不暴露。`/api/state` 做跨设备内存中转。
- **LLM 网关**：TAL GPT-5.5（OpenAI 兼容），凭证在 `饮食B段/.env`（`TAL_MLOPS_APP_ID` / `TAL_MLOPS_APP_KEY` / `TAL_MODEL`）。图片走 TAL gpt-image-1.5（鉴权头用 `api-key: appId:appKey`，与 chat 的 `Authorization: Bearer` 不同）。
- **跨设备同步**（`src/sync.ts`）：启动拉后端快照覆盖本地种子 → 本地变更 debounce 400ms POST → 每 1.5s 轮询版本号拉取。同步内存态，dev server 重启清空。
- **测试时间模拟**（`TimeControl.tsx`）：设备端与家长端共享模拟时钟，可设日期/时间或预设按钮；48h 归档以模拟时钟判定。

## 设计护栏（改代码时必须守住）

1. **只定性，不定量**：全程不输出克数/卡路里/蛋白质克数/营养缺口；参照《中国学龄儿童膳食指南》餐盘建议做序数判断。
2. **拍照链路三类入口统一**：被动 eating / 二次确认 / 主动拍照进同一条 引导+倒数+自动拍+判质量+补拍+降级 链路。
3. **零食不做独立入口/独立流程**：与正餐同走识别链路，按识别出的食物类别自然分流；餐盘结构只看蔬菜/主食/肉蛋豆（零食不在其中自然不参与）；劝导按识别出糖/膨化/含糖饮料触发，无需「零食记录」前提。
4. **频次护栏**：营养小科普每正餐≤1、每天≤3、同食物当天不重复；零食劝导每天≤1；补拍不另算打扰、不刷新冷却。
5. **双记录保留**：AI 原始识别 + 家长确认值并存，修改不覆盖原始。
6. **低置信度交给家长**：标记「待确认」按时间归入对应餐次；48h 未处理自动归档不计趋势。
7. **趋势口径 = confirmed**；**单次异常不焦虑**（周报只看趋势，连续偏低≥3 天才高亮）。
8. **硬护栏 LLM 不可越过**：抬腕唤醒门、30min 冷却、同周期≤2 次主动提醒、触发门槛 P≥70%（主动记录除外）、仅高置信度照片计趋势。

## 协作约定（用户偏好，务必遵守）

- **一次修改，不懂先问**：用户给一串需求时，一次性全做完；对任何不清楚的点，先问，不要默默替用户选一个解释。不交半成品。
- **最小化特殊分支**：设计/方案遇到边界情况，先问「能否用一条统一规则自然覆盖」，而不是默认为每个变体单独建模独立入口/类型/流程。衔接层尤其要薄——不在交接处做下游分流，分流留给真正执行识别的模块。只有当统一规则真的不能覆盖、且有强业务理由时，才引入特殊分支。
- **PRD 是真相源**：改行为前对照 PRD，存疑时读 PRD 而非猜代码。
